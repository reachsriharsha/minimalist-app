# Design: External REST functional test suite

## Approach

Add a **separate uv project** under `tests/` whose only purpose is to fire HTTP requests at a running compose stack and assert the documented contract. Add a small POSIX `bash` driver `test.sh` at the repo root that sequences: (1) readiness probe, (2) optional `make up`, (3) `uv run pytest` inside `tests/`, (4) optional `make down`. No production code is imported вҖ” tests see only what an external client sees.

Key design decisions (rationales inline):

- **Separate uv project vs. backend workspace member.** `tests/` is intentionally not a member of `backend/pyproject.toml`. Two reasons: (a) Semantically, these tests are an *external* consumer of the backend's public API вҖ” same role a third-party SDK's tests would play вҖ” so coupling them into the backend's dependency graph misrepresents the architecture. (b) Operationally, users who vendor this scaffold will often swap the backend language/runtime entirely (Go, Node, etc.) while keeping the test suite; a workspace member would make that swap harder. Cost of separation: `tests/uv.lock` is maintained independently, and there is no code-level import sharing with the backend (which is fine вҖ” there is nothing worth sharing).
- **Stack-lifecycle default = bring up if not healthy, leave running.** The common developer loop is "iterate on code, re-run tests." Tearing the stack down after each run would add 30вҖ“60 s of rebuild/startup on every cycle for zero benefit. The `--down` flag exists for the "I want a clean slate" or "CI one-shot" case; `--no-up` exists for "I already have a stack I want to protect from `make up --build` churn." These three modes cover local dev, CI, and surgical re-runs.
- **Readiness wait = fixed 1 s interval, 60 s total timeout.** Exponential backoff was considered and rejected: the readiness window we care about is 1вҖ“30 s (compose warm-up), an exponential schedule either fires too often early or waits too long late. A fixed 1 s tick gives predictable, readable output (`[test.sh] waiting for /readyzвҖҰ 5s`). 60 s is generous for cold-start on a laptop; if it is exceeded, something is genuinely wrong and the caller should see a fast, clear failure.
- **Test isolation = tolerate state, don't manufacture it.** Redis has no per-test reset and there is no DB snapshot mechanism in this scaffold. Tests that depend on counters (`hello_count`) assert **relative** behaviour вҖ” "the second call's `hello_count` is exactly one greater than the first" вҖ” not absolute values. This keeps the suite correct on re-runs, on a fresh stack, and on a stack that has served other traffic. Users who clone the template and need stricter isolation can add their own fixture (a Redis `FLUSHDB` in a setup hook, or a per-test DB transaction rollback); that is not this feature's job.

## `test.sh` behaviour in detail

Expected shape:

```bash
#!/usr/bin/env bash
set -euo pipefail

# Flags: --down (teardown after tests), --no-up (skip bring-up)
# Env:   TEST_BASE_URL (default http://localhost:8000)
#        READINESS_TIMEOUT (default 60, seconds)

repo_root="$(cd "$(dirname "$0")" && pwd)"
base_url="${TEST_BASE_URL:-http://localhost:8000}"
readiness_timeout="${READINESS_TIMEOUT:-60}"
do_up=1
do_down=0
for arg in "$@"; do
  case "$arg" in
    --no-up) do_up=0 ;;
    --down)  do_down=1 ;;
    -h|--help) # print usage and exit 0
      ;;
    *) echo "[test.sh] unknown argument: $arg" >&2; exit 2 ;;
  esac
done

probe_ready() {
  curl -fsS -o /dev/null -m 2 "${base_url}/readyz"
}

if ! probe_ready; then
  if [ "$do_up" -eq 1 ]; then
    echo "[test.sh] stack not ready at ${base_url}; running 'make up'..."
    (cd "$repo_root" && make up)
  else
    echo "[test.sh] stack not ready at ${base_url}; --no-up set, not bringing it up" >&2
    # still enter the wait loop below; will time out and exit non-zero
  fi

  deadline=$(( $(date +%s) + readiness_timeout ))
  while ! probe_ready; do
    if [ "$(date +%s)" -ge "$deadline" ]; then
      echo "[test.sh] readiness timeout after ${readiness_timeout}s" >&2
      exit 1
    fi
    sleep 1
  done
fi

echo "[test.sh] stack ready at ${base_url}; running tests..."
rc=0
(cd "${repo_root}/tests" && TEST_BASE_URL="${base_url}" uv run pytest) || rc=$?

if [ "$do_down" -eq 1 ]; then
  echo "[test.sh] --down set; running 'make down'..."
  (cd "$repo_root" && make down) || true  # teardown failure must not overwrite pytest rc
fi

exit "$rc"
```

(Above is indicative, not normative вҖ” Vulcan may restructure as long as the contract in "Acceptance Criteria" of the feature spec holds.)

Notes:

- `set -euo pipefail`: any unexpected error or unset variable fails fast.
- Teardown errors are swallowed *only* in the `--down` branch so that a teardown hiccup cannot mask test failure.
- `curl -fsS -m 2`: 2-second per-request timeout on the readiness probe; fails silently on non-2xx so the loop can retry cleanly.
- `make up` is idempotent (compose is), so a redundant call when the stack is already healthy is cheap вҖ” but the `probe_ready` short-circuit avoids even that.

## `tests/conftest.py` design

Fixtures:

- `base_url` (session-scoped): returns `os.environ.get("TEST_BASE_URL", "http://localhost:8000")`.
- `client` (session-scoped): yields an `httpx.Client(base_url=base_url, timeout=10.0)`; closes on teardown.
- `readiness_check` (session-scoped, autouse): on first use, does a single `GET /readyz`; if it fails or returns non-200, raises `pytest.UsageError` with a message of the form `"stack not reachable at {base_url}; did you run ./test.sh or `make up`?"`. This is a defence-in-depth check for the case where a developer runs `uv run pytest` directly without going through `test.sh`.

No fixtures reset state. No fixtures import from `backend.app`.

## Edge Cases & Risks

| Risk | Mitigation |
|---|---|
| Stack is up but one dependency (e.g. Redis) is unhealthy | `/readyz` returns 503 when `checks.redis != "ok"`; the readiness probe in `test.sh` uses `curl -f`, which fails on 503. The loop retries until timeout, then exits non-zero with a clear message. The subsequent `test_readyz` assertion makes the failure reason visible in pytest output too. |
| `hello_count` wraps / resets mid-test (Redis flush, container restart) | Low probability during a single test session; mitigated by fetching both values inside the same test function with a single client. If it ever bites, the fix is to retry once вҖ” deferred until observed. |
| Clock drift between host and containers | Not relied on. The suite does not assert timestamps. |
| `X-Request-ID` case mismatch | `httpx` normalizes response headers to case-insensitive lookup; assertions use `response.headers.get("x-request-id")` which handles either casing on the wire. |
| `uv` not installed on the host | `test.sh` will emit a shell-level "command not found" from `uv run`. `tests/README.md` lists `uv` as a prerequisite alongside Docker. Installation pointer: `curl -LsSf https://astral.sh/uv/install.sh \| sh`. |
| `make up --build` is slow on first run, tests time out waiting | The 60 s readiness timeout begins *after* `make up` returns. If users on slow machines hit it, they can export `READINESS_TIMEOUT=180`. Documented in `tests/README.md`. |
| Someone runs `./test.sh` from a different working directory | `test.sh` resolves its own directory via `"$(cd "$(dirname "$0")" && pwd)"` and uses an absolute `repo_root` for all subsequent `cd` calls. |
| Host port 8000 is bound by a *different* service (not our backend) | `/readyz` against it returns non-200 or a connection error; readiness probe fails; `make up` either succeeds (if the port is ours and we just hadn't started yet) or fails with a clear `port already allocated` from compose. Either way, the failure is loud. |
| Tests run against a stale stack that predates a backend code change | This is user error, not a bug in the harness. Documented in `tests/README.md`: when in doubt, `./test.sh --down && ./test.sh`. |
| Teardown race: `--down` called while a test is mid-request | `test.sh` runs `make down` only *after* `uv run pytest` has returned, so no such race exists within a single invocation. |

## Dependencies

External tooling (already required by earlier features; this feature adds no new external dependencies to the repo):

- `uv` вҖ” Python project/package manager. Pre-existing, used by `feat_backend_001`.
- `docker` + `docker compose` вҖ” pre-existing, used by `feat_infra_001`.
- `make` вҖ” pre-existing, used by `feat_infra_001`.
- `curl` вҖ” used by the readiness probe in `test.sh`. Ubiquitous; documented as a prerequisite.
- `bash` вҖ” `test.sh` is bash, not pure POSIX `sh`, because `set -euo pipefail` and `[[ ]]`-style constructs are clearer and these are bootstrap-template conventions.

Python packages (pinned in `tests/uv.lock`):

- `pytest >= 8.0` (matches backend's dev dependency).
- `httpx >= 0.27` (matches backend's dev dependency).

No other Python packages. Deliberately minimal.
