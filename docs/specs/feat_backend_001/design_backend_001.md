# Design: FastAPI backend scaffold

## Approach

Implement a single Python package `app` under `backend/app/` using standard FastAPI idioms: a module-level `FastAPI` instance created by a factory (`create_app`), lifespan-managed connection pools for Postgres and Redis, dependency-injected sessions and clients, a versioned router under `/api/v1`, and two operational endpoints (`/healthz`, `/readyz`) mounted at the app root.

Dependency management uses `uv`: a `backend/pyproject.toml` declares the project and dependencies, `uv` resolves and locks to `backend/uv.lock`, and a virtualenv is materialized under `backend/.venv/` by `uv sync`. All commands run from `backend/`; `app.*` imports work because `uv` puts the project's source root on `sys.path` via its project layout.

Migrations use Alembic in async mode: `alembic/env.py` builds an async engine from the same settings object the app uses, and a single initial revision creates an `items` table. The `items` table is seeded with exactly one row (`id=1`, `name="hello"`) inside the migration's `upgrade()` function so the hello endpoint has something to read on a fresh database.

Logging uses `structlog` configured to emit single-line JSON to stdout. An ASGI middleware assigns a request ID (either echoing the incoming `X-Request-ID` header or minting a new UUID4), binds it into `structlog.contextvars`, writes it on the response, and clears it on the way out. A global exception handler converts any uncaught exception into the documented error envelope.

Settings come from `pydantic-settings` reading `backend/.env`, with environment variables overriding file values. All settings have safe dev defaults so the app imports even with no `.env`.

Tests are a **small** pytest suite at `backend/tests/` for the in-container developer loop: an `httpx.AsyncClient` fixture against the ASGI app, smoke tests for `/healthz`, `/readyz`, and `/api/v1/hello`, and guard tests that skip when a live DB/Redis is not reachable. This suite is not the template's external contract — `feat_testing_001` owns that.

## Key types and interfaces

### Settings (`app/settings.py`)

```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8", extra="ignore")

    app_name: str = "minimalist-app-backend"
    env: Literal["dev", "test", "prod"] = "dev"
    host: str = "0.0.0.0"
    port: int = 8000
    log_level: str = "INFO"

    database_url: str = "postgresql+asyncpg://postgres:postgres@localhost:5432/app"
    redis_url: str = "redis://localhost:6379/0"

    request_id_header: str = "X-Request-ID"
```

`get_settings() -> Settings` is `@lru_cache`-wrapped so a single instance is reused.

### Error envelope (`app/errors.py` + `app/schemas.py`)

Every error response — whether from `HTTPException`, validation, or an unhandled exception — returns this JSON shape:

```json
{
  "error": {
    "code": "internal_error",
    "message": "Human-readable summary.",
    "request_id": "7a6f...-..."
  }
}
```

- `code` values: `http_error` (for `HTTPException`), `validation_error` (for `RequestValidationError`), `internal_error` (for any other `Exception`). Additional domain-specific codes may be added later.
- `message` is the exception's `detail` for HTTP errors, a short string for validation errors (with full details available in logs), and `"Internal Server Error"` for unhandled exceptions — **do not leak tracebacks to clients**. Tracebacks are logged at error level.
- `request_id` is the same ID on the `X-Request-ID` response header.

### Response shapes

```python
class HealthResponse(BaseModel):
    status: Literal["ok"]

class DependencyCheck(BaseModel):
    db: str      # "ok" or error string
    redis: str   # "ok" or error string

class ReadinessResponse(BaseModel):
    status: Literal["ready", "not_ready"]
    checks: DependencyCheck

class HelloResponse(BaseModel):
    message: str        # e.g. "hello from minimalist-app"
    item_name: str      # read from Postgres
    hello_count: int    # Redis counter value *after* increment
```

### Request ID middleware

Pure ASGI middleware (not `BaseHTTPMiddleware` — that one has issues with streaming and background tasks). Pseudocode:

```
async def __call__(self, scope, receive, send):
    if scope["type"] != "http": return await self.app(scope, receive, send)
    headers = MutableHeaders(scope=scope)
    rid = headers.get(self.header_name) or str(uuid4())
    token = structlog.contextvars.bind_contextvars(request_id=rid)
    async def send_wrapper(message):
        if message["type"] == "http.response.start":
            message["headers"].append((self.header_name.lower().encode(), rid.encode()))
        await send(message)
    try:
        await self.app(scope, receive, send_wrapper)
    finally:
        structlog.contextvars.unbind_contextvars("request_id")
```

### DB and Redis lifespans

`create_app` attaches a `lifespan` context manager:

- On startup: build the SQLAlchemy async engine from `settings.database_url`; build the Redis client from `settings.redis_url`. Stash both on `app.state`.
- On shutdown: `await engine.dispose()` and `await redis.aclose()`.

FastAPI dependencies pull from `app.state`:

```python
async def get_session(request: Request) -> AsyncIterator[AsyncSession]:
    async with request.app.state.sessionmaker() as session:
        yield session

async def get_redis(request: Request) -> Redis:
    return request.app.state.redis
```

## Config surface (`.env.example`)

```
# Application
APP_NAME=minimalist-app-backend
ENV=dev
HOST=0.0.0.0
PORT=8000
LOG_LEVEL=INFO

# Postgres (async driver required)
DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5432/app

# Redis
REDIS_URL=redis://localhost:6379/0

# Request ID
REQUEST_ID_HEADER=X-Request-ID
```

Every variable in this file is read by `Settings`. Adding a new setting means updating both `Settings` and `.env.example` in the same commit.

## Data Flow

**`GET /api/v1/hello`:**

1. `RequestIDMiddleware` assigns/echoes `X-Request-ID` and binds it into log context.
2. FastAPI resolves dependencies `get_session` and `get_redis`.
3. Handler runs `SELECT name FROM items WHERE id = 1` via `AsyncSession`.
4. Handler runs `redis.incr("hello:count")`, capturing the new value.
5. Handler returns `HelloResponse(message="hello from minimalist-app", item_name=..., hello_count=...)`.
6. FastAPI serializes to JSON; middleware appends `X-Request-ID` header to the response.

**`GET /readyz`:**

1. Handler opens a session, executes `SELECT 1`; records `"ok"` or the exception's str.
2. Handler calls `await redis.ping()`; same treatment.
3. If both are `"ok"`, return `200 ReadinessResponse(status="ready", ...)`; otherwise return `503` with the same body shape but `status="not_ready"`.
4. Errors during probes do **not** raise — they are captured into the checks dict.

**Startup:**

1. `configure_logging()` runs first so every log after this point is JSON.
2. `create_app` builds the engine, sessionmaker, Redis client, attaches to `app.state`.
3. App yields to ASGI server; on shutdown, pools are disposed.

## Edge Cases & Risks

| Risk / edge case | Mitigation |
|---|---|
| Postgres or Redis is down at startup. | Do not connect at import time. The engine and Redis client are *constructed* (cheap, no network) at startup; actual connections happen lazily on first use. `/healthz` never touches them. `/readyz` surfaces the failure as a 503. |
| `DATABASE_URL` uses the sync driver (`postgresql://` without `+asyncpg`). | `Settings` accepts the string as-is, but `app.db` validates at engine construction that the URL's driver is async; if not, raise a clear error with the correct format in the message. |
| Alembic `env.py` in async mode is notoriously easy to get wrong. | Use the official async template pattern: `async def run_async_migrations()` that calls `connection.run_sync(do_run_migrations)`; run it via `asyncio.run(...)` inside `run_migrations_online()`. |
| Seed row (`items.id=1`) already exists when migration re-runs on a non-empty DB. | Only the initial migration inserts; downgrade drops the table. Idempotent re-seeding is not a goal for the scaffold. |
| `BaseHTTPMiddleware` subtle bugs around streaming responses and background tasks. | Use a plain ASGI middleware for `RequestIDMiddleware`, not `BaseHTTPMiddleware`. |
| `structlog` and stdlib `logging` diverging. | `configure_logging()` routes stdlib logging through `structlog`'s `ProcessorFormatter` so FastAPI/uvicorn logs also emit JSON. |
| Unhandled exceptions leaking stack traces to clients. | Global `Exception` handler returns the fixed error envelope with a generic `message`; full traceback is logged at error level, not returned. |
| Tests accidentally becoming the template contract. | `backend/tests/` stays minimal; its README-style docstring in `conftest.py` states this is a dev-loop suite; `feat_testing_001` owns the external contract. |
| Test suite fails when run with no DB/Redis. | Tests that need them use a `require_db` / `require_redis` fixture that calls `pytest.skip` with a clear message on connection failure. `/healthz` test always runs. |
| Committing `.env` by accident. | `backend/.gitignore` includes `.env`. `.env.example` is the committed template. |
| Port collision with other local services. | `PORT` is configurable via env; default `8000` documented in `.env.example`. |
| Later features (`feat_infra_001`) need to override settings from docker-compose. | All settings are env-var driven, no hard-coded values in source. `pydantic-settings` reads env vars with priority over `.env`, so compose can set them directly. |
