# Test: Backend rules and logging discipline

## Scope

This feature is a documentation artifact (`RULES.md`), a refactor (`items` domain move), and two small structural additions to the logger (`CallsiteParameterAdder`, `redact_sensitive`). External behavior is unchanged. Test coverage mirrors what is actually verifiable at the code level.

## New tests

### `backend/tests/test_logging_callsite.py`

Verifies the `CallsiteParameterAdder` addition. Runs against the configured JSON handler to ensure real output carries the new fields.

| Case                          | Arrangement                                                                 | Assertion                                                                 |
|-------------------------------|-----------------------------------------------------------------------------|---------------------------------------------------------------------------|
| Info log carries callsite     | Call `configure_logging`; emit `log.info("probe", detail="x")` from a known line. | Captured JSON contains `filename`, `func_name`, `lineno`; `lineno` matches the known line. |
| Exception log carries trace   | `configure_logging`; inside a `try/except`, call `log.exception("failed", op="probe")`. | Captured JSON has `event == "failed"`, carries `exception` (or `stack_info`) string containing the raising line's function name, and `lineno` of the `log.exception` call. |

Capture strategy: attach a `logging.Handler` subclass that appends each rendered JSON string to a list, rather than reading from `sys.stdout`. Decode the JSON in the assertion.

### `backend/tests/test_logging_redaction.py`

Verifies the `redact_sensitive` processor. Direct unit test of the processor function; no logger instance needed.

| Case                              | Input event_dict                                          | Expected output                                            |
|-----------------------------------|-----------------------------------------------------------|------------------------------------------------------------|
| Denylisted top-level key          | `{"password": "hunter2", "user_id": 7}`                    | `{"password": "***", "user_id": 7}`                        |
| Non-denylisted top-level key      | `{"item_id": 42, "item_name": "widget"}`                   | Identical to input.                                        |
| Denylisted key in nested dict     | `{"request": {"authorization": "Bearer x", "path": "/"}}` | `{"request": {"authorization": "***", "path": "/"}}`      |
| Denylisted key in deeply nested dict | `{"a": {"b": {"password": "p"}}}`                      | `{"a": {"b": {"password": "***"}}}`                        |
| Mixed-case denylisted key         | `{"Authorization": "Bearer x"}`                           | `{"Authorization": "***"}`                                 |
| Non-string key                    | `{1: "ok", "token": "t"}`                                 | `{1: "ok", "token": "***"}`                                |
| List containing dict (not traversed) | `{"users": [{"password": "p"}]}`                        | Identical to input. Documented as a known limitation.      |

### `backend/tests/test_hello.py` (updated, not new)

Existing test. Only the import paths change (from `app.api.v1.hello` / `app.models` / `app.schemas` to `app.items.router` / `app.items.models` / `app.items.schemas`). Assertions are unchanged. If the response body ever diverges from the pre-feature version, this test fails and blocks the build.

## What is intentionally not tested

- **`RULES.md` content.** A document, not code. Proofreading is part of PR review.
- **Domain folder layout.** Verified transitively: if any file is misplaced, imports break, and the whole test suite fails on collection.
- **Alembic naming convention.** Asserting `Base.metadata.naming_convention == {...}` is tautological — it repeats the production code in a test. The convention is exercised the next time a migration is autogenerated; until then, there is nothing to verify.
- **Alembic `file_template`.** Same reason. The slug template applies to filenames generated by `alembic revision`; there is no runtime path to cover.
- **The "every log line carries a domain kwarg" rule.** Rule only, not enforced. No test.
- **The "`async def` only when awaiting I/O" rule.** Not mechanically verifiable without a linter. Deferred with the rest of linting.
- **Existing middleware / error envelope / request-ID behavior.** Unchanged by this feature. Covered by the existing suite.

## Regression surface

| Regression                                               | What fails                                              |
|----------------------------------------------------------|---------------------------------------------------------|
| `CallsiteParameterAdder` dropped or misconfigured        | `test_logging_callsite.py`                              |
| Redactor weakened (e.g., drops nested traversal)         | `test_logging_redaction.py` nested-dict case            |
| Redactor made case-sensitive                             | `test_logging_redaction.py` mixed-case case             |
| Shared schema accidentally moved into `app/items/`       | Import in `app.errors` or `app.api.health` breaks; full suite fails on collection. |
| `Base` left in `app/models.py` and not in `app/db.py`    | Import in `alembic/env.py` or `app.items.models` fails; full suite fails on collection. |
| Response body for `GET /api/v1/hello` changes            | `test_hello.py`                                         |

## Security notes

- Tests do not put real secrets into log events. The `password` and `token` values used as test fixtures are literal strings like `"hunter2"` and `"Bearer x"`.
- The redactor is verified structurally; it is not a substitute for the documented rule "never log secrets." `RULES.md` carries the rule; the test carries the safety net.
- No test writes to the filesystem, network, or external services. Tests run in the standard backend `pytest` suite.
