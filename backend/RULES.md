# Backend Rules

This file is the shaping guide for every backend feature that lands after `feat_backend_002`. It is read by humans when reviewing PRs and by the AutoDev agents (Atlas, Vulcan) when planning and generating code. It encodes a trimmed, template-scaled subset of the well-known fastapi-best-practices conventions, scoped to what actually pays off at scaffold size.

The rules are split across six sections. Each rule is a single declarative sentence followed by a one-line reason. No rule appears without its *why*. If a rule ever needs to change, update this file in the same PR that changes the behavior — never let the document drift from the code.

## 1. Layout

- **Domain folder per business resource.** Each resource lives at `app/<domain>/{router,schemas,models,service}.py`, with `dependencies.py`, `constants.py`, and `exceptions.py` added only when the file would otherwise be non-trivial. *Why: a feature's entire surface area stays readable inside one folder.*
- **Shared infra stays flat at `app/`.** `settings.py`, `db.py`, `redis_client.py`, `logging.py`, `middleware.py`, `errors.py`, and `main.py` are not owned by any one domain and never move into a subfolder. *Why: these modules are cross-cutting; burying them inside a domain would lie about ownership.*
- **Absolute imports only.** Always write `from app.items.service import ...`, never `from .service import ...`. *Why: absolute paths survive file moves and copy-paste; relative paths silently break when a module is relocated.*

## 2. API

- **Every business route mounts under `/api/v<N>/`.** Health and readiness probes (`/healthz`, `/readyz`) stay unversioned. *Why: versioning is a guarantee to external clients; infrastructure probes are not part of that contract.*
- **Every route declares `response_model=...` and an explicit `status_code` when non-200.** *Why: OpenAPI docs and downstream client generators rely on both to produce accurate types.*
- **URLs are RESTful: plural nouns, no verbs in paths.** Validation lives in pydantic schemas and `Depends(...)` helpers, not inside handler bodies. *Why: handlers stay thin and service code stays framework-agnostic, so swapping web frameworks or reusing a service in a worker is a mechanical edit.*
- **Use `async def` only when the route awaits I/O.** Pure-CPU or blocking work uses `def` so Starlette runs it in a threadpool. *Why: a blocking call on an `async def` route blocks the entire event loop and freezes every other concurrent request.*

## 3. Data layer

- **SQLAlchemy 2.x `select()` style only.** No legacy `Query` API. Writes wrap in `async with session.begin(): ...` or perform an explicit `await session.commit()`. *Why: the legacy API is deprecated, and the mixture of styles is a common source of silently-uncommitted writes.*
- **Alembic metadata uses a naming convention for constraints.** Names follow `pk_<t>`, `fk_<t>_<c>_<ref>`, `uq_<t>_<c>`, `ix_<t>_<c>`, `ck_<t>_<name>`, set on `Base.metadata.naming_convention` in `app/db.py`. *Why: without a convention, autogenerated migrations produce database-dependent names that differ between Postgres versions and block downgrade.*
- **Migration filenames are descriptive.** `alembic.ini`'s `file_template` is set to `%%(rev)s_%%(slug)s` so `alembic revision -m "add item owner"` yields `0042_add_item_owner.py`. *Why: `0042_add_item_owner.py` is greppable and self-documenting; `0042_abcdef.py` is not.*

## 4. Settings and secrets

- **All config reads go through `app.settings.Settings`.** No `os.environ[...]` calls anywhere else in the package. *Why: a single module owns defaults, validation, and documentation — readers do not have to chase env names across the codebase.*
- **Every new setting lands in `settings.py` and `.env.example` in the same commit.** *Why: `.env.example` is the onboarding document; a setting that exists in code but not in the example is invisible to new contributors.*
- **No secrets in the repo.** `backend/.env` is gitignored; `backend/.env.example` holds safe development defaults only, never production credentials. *Why: git history is forever — a leaked secret cannot be recalled by deleting the file later.*

## 5. Logging

- **Every log line carries `timestamp`, `level`, `event`, `filename`, `func_name`, `lineno`, and `request_id` (when in a request context).** The logger config in `app/logging.py` enforces this; callers do not add these fields by hand. *Why: a log without a source location is useless during an incident.*
- **Every log line should carry at least one domain kwarg** (`item_id=...`, `user_id=...`, `order_id=...`). This is documented as a rule but not enforced at runtime. *Why: forcing required kwargs tends to push callers toward `log.info("x", **user.__dict__)` style dumps that leak PII; we teach the rule instead of ramming it through a processor.*
- **On caught exceptions, use `log.exception("event_name", **data)` or `log.error("event_name", exc_info=exc, **data)`.** Never stringify the exception into the event name or into a free-form message. *Why: `exc_info` preserves the full traceback through the structlog chain; a stringified exception loses every frame.*
- **Never log secrets or PII.** The logger redacts a fixed denylist (`password`, `authorization`, `token`, `cookie`, `secret`, `api_key`, `set-cookie`, `refresh_token`) case-insensitively at every dict depth. Extending the denylist is a one-line change in `app/logging.py`. *Why: a mistake at a single call site must not be able to leak a credential; the redactor is defense in depth.*
- **Event names are `snake_case` noun phrases** (`item_fetched`, `redis_incr_failed`, `seed_item_missing`). Not full sentences, not verbs. *Why: event names are indexed and searched; consistent shapes are easy to filter and aggregate.*
- **No `print()`.** Anywhere in `app/` or `tests/`. *Why: `print` bypasses the structured formatter and the redactor, producing raw bytes to stdout that break log aggregators and can leak secrets.*

### Known redactor limitations

- Lists, tuples, and sets are not traversed. A log line that passes a list of user dicts (e.g., `log.info("batch_done", users=[{...}, {...}])`) will not be redacted inside the list. Do not log collections of user-shaped dicts; log counts and IDs instead.
- Values that have been pre-rendered into strings (including stringified JSON bodies or `repr()` output) are not scanned. Structure your log kwargs as dicts and let the formatter render them.
- The denylist matches by key, not by value. A secret stored under a non-denylisted key name (e.g., `{"blob": "hunter2"}`) is passed through verbatim. Name your kwargs honestly.

## 6. Testing

- **`pytest` + `httpx.AsyncClient` + `pytest-asyncio`.** Tests that need Postgres or Redis skip (with a clear reason) when the dependency is unreachable; the backend pytest suite must succeed on a bare checkout with no services running. *Why: a hard dependency on external services blocks developer iteration and CI without a local stack.*
- **One test file per domain router; shared fixtures in `backend/tests/conftest.py`.** *Why: a failing test immediately points at its domain without grepping across files.*
